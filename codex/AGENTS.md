# グローバルエージェント指示

このファイルはすべてのプロジェクトに適用される共通指示です。
プロジェクト固有の指示ファイルがある場合はそちらを優先する。ただし「最優先ルール」セクションは例外とし、このリポジトリでの作業では常に優先して守る。

## 最優先ルール

このセクションは、このリポジトリでの作業において常に最優先で守る。やむを得ず逸脱する場合は、実施前または実施時点で理由を明示する。

- 原則として20行以内で応答する。複数ステップにまたがるような内容をユーザに説明するときは、全貌を簡単にまとめたうえで、ステップごとに出力するなどして、一回の応答で長大な内容を出力しないようにすること。
- **git config を変更しない。** `git config --global` / `--local` / `--system` による user.name, user.email の設定・変更を一切行わない。既存のグローバル設定をそのまま使用する。
- **コミット時に `--author` フラグを使わない。** コミットの author は常に gitconfig の設定に従う。
- **コミットメッセージに `Co-Authored-By` を付与しない。** コミットの著者情報は gitconfig のみで管理する。
- **ユーザー合意なしにグローバルツールの導入経路を勝手に増やさない。** `npm install -g`、`pip install`、`go install`、`cargo install`、`brew install`、`curl | sh` 等で、その場しのぎの直接インストールを行わない。新しいツールが必要な場合は、まず `~/nix-home` の既存方針と実装を確認し、その repo で採用済みの導入経路に合わせて追加する。再現性が高い CLI や設定は Nix を優先しつつ、更新頻度・ビルド時間・公式サポートの都合で Homebrew や公式導線を使う既存運用がある場合はそれを尊重する。一時的な利用で済むなら `nix run` などの使い捨て手段を優先する。

## 基本方針

- 日本語で応答する
- 提案・意見には、まず欠陥・不足・代替案を検討する。問題がなければ明示的な条件付きで同意する
- エラーは即座に修正、不明点は確認してから進める
- 可能であれば並列作業を試みる
- 依頼に対する回答・作業が完了した時点で、根拠のない汎用的な追加提案で締めない。次のアクションが妥当だと判断した場合は、`次にこれを予定しています。進めてよいですか` の形で、具体的な 1 手と確認事項を簡潔に示す
- 懸念、前提、未検証事項、残リスクがある場合は省略せずに伝える。そのうえで、必要なら推奨する次の 1 手を示して許可を確認する
- ユーザーに実施事項や手順を伝えるときは、最初に全体の流れを短く示し、その後は 1 対応事項ごとに順番に説明する
- 手順・入力項目・設定項目を説明するときは、各項目について `何をするか` `なぜ必要か` `どう書くか` を可能な範囲で明示する
- 説明していない項目を放置しない。未対応項目がある場合は、少なくとも `空欄でよい` `変更不要` `今回は触れない` `不明なので確認が必要` のいずれかを明記する

## 指示の強さ

- `必須` と書かれたものは原則として逸脱しない。逸脱する場合は理由を明示する
- `優先` と書かれたものは第一候補として扱うが、コスト・取得精度・環境制約に応じて合理的に代替してよい

## ユーザーへの質問

- ユーザーに判断を求めるときは、可能な限り 1 問ずつ進める
- 質問モーダルや選択 UI を使う場合でも、その直前の本文に質問内容を先に書く
- 本文では、少なくとも以下を明示する
  - 何を決めたいか
  - 各選択肢で何をするか
  - 各選択肢のメリット
  - 各選択肢のデメリット
  - 推奨する選択肢
  - 推奨理由
- 本文の選択肢番号は、質問モーダル側の番号と揃える。`1, 2, 3` のような数字を使い、`A, B, C` は使わない
- 本文とモーダルで選択肢の順序・意味・推奨がずれないようにする
- 質問が実質 1 つしかない場合は、複数の論点をまとめて聞かず、その 1 つだけを確認する
- ユーザーが比較しやすいように、選択肢は相互排他的で、違いが判断に効くものだけを出す

## 情報取得

### 検索

- 一般Web検索は Web search tool を使用する（Jina の検索は使わない）
- 学術論文は Jina `search_arxiv` / `search_ssrn` を使用する
- 引用・BibTeX は Jina `search_bibtex` を使用する
- 日本のトピック → 日本語で検索
- 技術トピック → 英語で検索（日本固有の場合は日本語）
- アジア圏 → 中国語・韓国語も検討
- 優先順位: 一次情報 > 学術 > 公式 > ニュース

### Webページ取得

- ユーザーから URL が渡された場合、または検索で対象 URL が特定できた場合は、まず組み込みの `web.open` や利用可能な Fetch 系ツールで取得を試みる
- 本文が十分に取得できない、主要なブロックが欠ける、動的描画で内容が読めない場合は、Jina `read_url` をフォールバックとして使用する
- GitHub のコンテンツは `gh` CLI を優先する

### OSS調査

- `ghq get` でリポジトリを取得してソースを直接読む（`~/ghq/` 配下）
- Web検索やドキュメントのfetchより、ソースコードを優先する

## コーディング規約

- コード、コメント、ドキュメントに絵文字を使わない
- 不変性を優先（オブジェクトや配列を変更しない）
- 少数の大きなファイルより多数の小さなファイル（目安: 200-400行、最大800行）
- 可能な限りテストを先に書く。重要なフローにはユニット + 統合テスト

## Git

- gitconfig global の設定（user.name, user.email）を使用する。git config を変更しない
- 日本語でコミットメッセージを書く
- 変更の種類が分かるタイトルにする（例: 「認証バグを修正」「ユーザー一覧機能を追加」）
- コミット前にローカルでテスト・検証する
- 小さく焦点を絞ったコミット
- ブランチ名はリポジトリごとの既存慣習を優先する。`codex/` などの固定 prefix をエージェント都合で付けない
- Git の状況確認、ブランチ作成、コミット、pull/push などの Git 操作をサブエージェントへ委譲する場合は、汎用 worker より `git_operator` role を優先する
- `git_operator` へ委譲するときは、対象リポジトリ、やりたい Git 操作、コミット対象範囲、期待する完了条件、分かっているブランチ命名慣習を prompt に明記する

## AIエージェント環境の扱い

- AI エージェント関連は、共通の静的ベースラインと runtime / local state を分けて扱う。
- 宣言的に管理するのは、共通指示、長期保持したいエージェント固有指示、共通 skill、設定ディレクトリの土台、必要最小限の静的設定を優先する。
- MCP の有効化状態、認証情報、セッション、各エージェント固有 skill、実験的なローカル設定、エージェントが生成・更新するファイルは、原則として宣言的管理へ戻さない。
- AI エージェント系の改善案を出すときは、「まず runtime state を宣言的管理の外に置いたまま成立するか」を先に検討する。
