# GitHub Copilot 共通指示

このファイルは GitHub Copilot のローカル共通指示として使う。
プロジェクト固有の指示ファイルがある場合はそちらを優先する。

## 最優先ルール

- `git config --global` `--local` `--system` を変更しない。
- `git commit --author` を使わない。
- コミットメッセージに `Co-Authored-By` を入れない。
- `npm install -g` `pip install` `go install` `cargo install` `brew install` `curl | sh` などでグローバルツールを直接入れない。必要なツールは `~/nix-home` の設定を更新して Nix 経由で導入する。

## 基本方針

- 日本語で応答する。
- まず欠陥・不足・代替案を検討し、問題がなければ条件付きで同意する。
- 懸念、前提、未検証事項、残リスクは省略せずに伝える。
- 不明点の確認は本当に必要な場合だけ行い、可能なら合理的な仮定を置いて進める。
- 手順や設定を説明するときは、最初に全体の流れを短く示し、その後は各項目について `何をするか` `なぜ必要か` `どう書くか` を明示する。

## 情報取得

- 一般 Web 検索は組み込みの Web 検索を優先する。
- URL が特定できている場合は、まず組み込みの open / fetch 系手段で取得を試みる。
- 本文の主要ブロックが欠ける、動的描画で読めないなど取得に失敗した場合だけ Jina `read_url` をフォールバックとして使う。
- OSS 調査は Web ドキュメントよりソースコードを優先し、GitHub コンテンツは `gh` CLI が使えるなら優先する。

## コーディング規約

- コード、コメント、ドキュメントに絵文字を使わない。
- 不変性を優先し、オブジェクトや配列をむやみに変更しない。
- 少数の大きなファイルより、多数の小さなファイルを優先する。
- 可能ならテストを先に書き、重要なフローにはユニットテストと統合テストを用意する。

## Git

- gitconfig の既存 `user.name` `user.email` をそのまま使う。
- コミットメッセージは日本語で、変更の種類が分かる短いタイトルにする。
- コミット前にローカルでテスト・検証する。
- ブランチ名はそのリポジトリの既存慣習を優先し、エージェント都合の固定 prefix を勝手に付けない。

## AI エージェント環境

- AI エージェント関連は、静的ベースラインと runtime / local state を分けて扱う。
- Nix や dotfiles で管理するのは、共通指示、長期保持したい agent 固有指示、共通 skill、設定ディレクトリの土台、必要最小限の静的設定を優先する。
- MCP の有効化状態、認証情報、セッション、実験的なローカル設定、agent が生成・更新するファイルは、原則として宣言的管理へ戻さない。
